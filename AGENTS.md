# Bubble Shield — Agent Notes

A Fabric mod for **Minecraft 26.2** that adds a furnace-like "Bubble Shield Projector" block: insert fuel (and an optional upgrade core), open its GUI to whitelist players, pick 1 of 350 effects, and set the bubble diameter (8–200), shape (sphere/dome) and mode (defense/pulse/eco, plus an optional periodic effect-cycle toggle). The bubble is a server-authoritative shield with per-effect surface shaders, inside behaviors, guard styles, context reactivity, ambient sounds and per-effect in-bubble screen post-effects + HUD; projectiles shrink it until it breaks (30-min cooldown, halved at tier 2). Comparator/redstone integration and advancements included. See `README.md` for the feature/user-facing overview and the standard `./gradlew` commands.

## Cursor Cloud specific instructions

- **Toolchain:** Minecraft 26.2 requires **JDK 25** (Temurin 25 is installed at `/usr/lib/jvm/jdk-25.0.3+9`). The startup update script installs it and persists `JAVA_HOME` in `~/.bashrc`. If a non-login shell lacks it, run `export JAVA_HOME=/usr/lib/jvm/jdk-25.0.3+9 && export PATH=$JAVA_HOME/bin:$PATH` before any `./gradlew` command. The build targets/`release`s Java 25, so an older JDK will fail.
- **First build is slow:** `./gradlew build` downloads Gradle 9.5.1 and Minecraft 26.2 on first run (several minutes); later builds are ~15–30s. There is no system-wide `gradle` — always use the wrapper `./gradlew`.
- **Primary verification is headless gametests:** `./gradlew runGameTest` runs the full server-side suite — currently **80 `@GameTest`s across 13 classes** in `com.bubbleshield.gametest` (the runner reports 81 — it also runs Fabric API's own built-in gametest) (shield lifecycle, projectile interactions, whitelist barrier, menu, tiers/regen, flux capacitor drain/regen economy, comparator/redstone, dome geometry, guard/context, advancements, boss bar/shield naming, shield modes/effect cycle, owner color override, resonance linking/damage split, the /bubbleshield command + sculk game events + Resonant Core loot injection, the 350-effect/50-behavior catalogue invariants, EN+DE lang key parity, post-effect assets, NBT-load hardening/legacy-save handling). The vanilla runner batches 50 tests per environment and ticks a batch's tests in parallel; in-level mock players are created through `com.bubbleshield.gametest.MockPlayers.createUniqueMockPlayer` (unique per-call names + automatic end-of-test PlayerList sweep on pass/fail/timeout) — never through the deprecated `GameTestHelper.makeMockServerPlayerInLevel`, whose shared "test-mock-player" name made `getPlayerByName` cross-resolve concurrent tests' mocks (the historical `whitelist_remove_revokes_uuid` flake, now fixed). New gametest classes should still declare their own `test_environment` (see `ColorGameTests.ISOLATED_ENVIRONMENT` + `data/bubbleshield/test_environment/color.json`) instead of growing the shared default batch past 50 and reshuffling it. Also beware barrier reach across the batch grid: a huge shield's `expelBlockedPlayers` sweep (e.g. `advancementMaximalist`'s diameter-200 activation, radius 100 > the ~45-block structure spacing) teleports OTHER concurrent tests' parked mock players out of their bubbles, so tests raising very large shields must run in an isolated batch and deactivate the shield before succeeding. Tests needing more room than the default 8x8x8 structure can ship an SNBT template under `data/bubbleshield/gametest/structure/*.snbt` and reference it via `@GameTest(structure = "bubbleshield:<name>")` — a minimal `{DataVersion, size, data:[one air entry], entities:[], palette:["minecraft:air"]}` works because the runner air-clears the whole `size` box before placement (see `linking_arena.snbt`, 39x8x8; keep height 8 so radius-8 shield boundary crossings stay above the barrier ceiling the framework builds at the structure top). This is the main correctness gate — use it, not just `build`. `./gradlew runServer` also boots a headless dedicated server (`run/eula.txt` is already accepted; Loom's `configureTests { eula = true }` auto-accepts the gametest EULA).
- **Shader validation (hard pre-commit gate):** the mod ships **350 generated per-effect bubble surface shaders** (`src/client/.../shaders/bubble/fx_000.fsh`–`fx_349.fsh` + the shared `surface.vsh`) and **350 generated per-effect screen-fx shaders** (`src/main/.../shaders/screenfx/sfx_000.fsh`–`sfx_349.fsh`) — 701 GLSL files total. They are emitted by deterministic byte-stable generators; the workflow is: edit `tools/gen_surface_shaders.py` / `tools/gen_screen_shaders.py` / `tools/gen_post_effects.py` → regenerate (run the scripts) → `python3 tools/validate_shaders.py` → `./gradlew build` → `./gradlew runGameTest`; NEVER hand-edit generated `fx_*`/`sfx_*` files or `post_effect/effect_*.json` (regeneration overwrites them). `validate_shaders.py` inlines the vanilla `#moj_import` includes, runs `glslangValidator` over all 701 files (multiprocessing; ~4–5 s on this VM) and enforces the generated-shader invariants (byte-uniqueness, `[layer:*]` markers + `tools/surface_manifest.json` agreement, `FxConfig` block/JSON uniform-order agreement + `tools/screen_manifest.json` agreement incl. its classpath copy). It must exit 0 after ANY shader change — this is a hard gate, not a nicety: every per-effect surface pipeline is registered up front and precompiled on EVERY client resource load (estimated +0.5–2.5 s total, unverifiable headless), and any single invalid surface shader fails the whole client resource load with `RuntimeException("Failed to load required shader programs")`. Screen shaders compile lazily (first bubble entry) and merely log+drop on failure, but gate them identically. Gametests cannot see the client-only bubble shaders.
- **No GPU on this VM — the client cannot be visually verified.** `./gradlew runClient` needs real OpenGL and will not render here, so the bubble **surface shaders** and **in-bubble screen post-effects** cannot be seen/tested in Cursor Cloud. Verify shader/rendering changes by: (1) `glslangValidator` on the GLSL (installed; expand `#moj_import` includes from the vanilla client jar first — `tools/validate_shaders.py` does all of this), (2) structural diffing against vanilla 26.2 shaders, and (3) build + `runGameTest` (asset-existence + JSON-parse tests). Visual quality claims ("stunning"/"perfect"/"8K") are NOT certifiable here — only compile-validity and structural invariants are; glslangValidator also does not equal every GPU driver's compiler. State this limitation honestly rather than claiming visual verification.
- **26.2 is UNOBFUSCATED / Mojang official mappings (NOT Yarn).** Old Yarn-name tutorials are wrong here (e.g. it's `Creeper`/`CreativeModeTabEvents`, not `CreeperEntity`/`ItemGroupEvents`; `AbstractArrow` is in `...projectile.arrow`). The build uses the `net.fabricmc.fabric-loom` plugin with plain `implementation` deps (no `modImplementation`/`remapJar`). When unsure of an API, decompile with `./gradlew genSources` and read the sources jars under `.gradle/loom-cache/minecraftMaven/**/*-sources.jar` — always confirm signatures there before using them.
- **26.2 API shape gotchas** (verified in this codebase): block-entity NBT uses `ValueInput`/`ValueOutput` (not `CompoundTag` params); GUIs are extractor-based (`extractBackground`/`extractLabels`, not `renderBg`); world rendering is submit-based (`LevelRenderEvents.COLLECT_SUBMITS` + `submitCustomGeometry`; `WorldRenderEvents` does not exist); datapack folders are singular (`loot_table`, `recipe`, `tags/block`).
- **Gametests live in the main source set** (`com.bubbleshield.gametest.ShieldGameTests`, registered via the `fabric-gametest` entrypoint) with `fabricApi { configureTests { createSourceSet = false; eula = true } }` in `build.gradle`. Post-effect/screenfx assets are intentionally under `src/main/resources` (not `src/client/resources`) so the headless gametest server classpath can see them for the asset-existence test.
