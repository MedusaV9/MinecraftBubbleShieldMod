# Bubble Shield — Agent Notes

A Fabric mod for **Minecraft 26.2** that adds a furnace-like "Bubble Shield Projector" block: insert fuel (and an optional upgrade core), open its GUI to whitelist players, pick 1 of **840 effects** (ids 0..839; 120 inside behaviors x 7 variants; 60 surface + 28 screen technique families), and set the bubble diameter (8–200), shape (10 shapes: sphere/dome/cylinder/cube/diamond/ring/pyramid/lens/hourglass/star), beam style (8 rendered styles — storm/pulse/helix/prism/void/ember/runic/frost — plus None/Auto) and mode (defense/pulse/eco, plus an optional periodic effect-cycle toggle). The bubble is a server-authoritative shield with per-effect surface shaders, inside behaviors, guard styles, context reactivity, ambient sounds and per-effect in-bubble screen post-effects + HUD; projectiles shrink it until it breaks (30-min cooldown, halved at tier 2). Comparator/redstone integration and advancements included. See `README.md` for the feature/user-facing overview and the standard `./gradlew` commands.

## Cursor Cloud specific instructions

- **Toolchain:** Minecraft 26.2 requires **JDK 25** (Temurin 25 is installed at `/usr/lib/jvm/jdk-25.0.3+9`). The startup update script installs it and persists `JAVA_HOME` in `~/.bashrc`. If a non-login shell lacks it, run `export JAVA_HOME=/usr/lib/jvm/jdk-25.0.3+9 && export PATH=$JAVA_HOME/bin:$PATH` before any `./gradlew` command. The build targets/`release`s Java 25, so an older JDK will fail.
- **First build is slow:** `./gradlew build` downloads Gradle 9.5.1 and Minecraft 26.2 on first run (several minutes); later builds are ~15–30s. There is no system-wide `gradle` — always use the wrapper `./gradlew`.
- **Primary verification is headless gametests:** `./gradlew runGameTest` runs the full server-side suite — currently **98 `@GameTest`s across 15 classes** in `com.bubbleshield.gametest` (the runner reports 99 — it also runs Fabric API's own built-in gametest) (shield lifecycle, projectile interactions incl. anti-tunneling through the thin RING tube, whitelist barrier, menu, tiers/regen, flux capacitor drain/regen economy, comparator/redstone, dome geometry, the ten shield shapes' containment math + barrier integration, guard/context, advancements, boss bar/shield naming, shield modes/effect cycle, owner color override, resonance linking/damage split, the /bubbleshield command + sculk game events + Resonant Core loot injection, the 840-effect/120-behavior catalogue invariants incl. a full-catalogue particle-capture containment matrix (k-sigma spread corners, fly-toward dip, per-tick <=128 particle budget, MAX-context cells) and a frozen-rows (ids 0..349) golden SHA-256, EN+DE lang key parity, post-effect assets, NBT-load hardening/legacy-save handling). The vanilla runner batches 50 tests per environment and ticks a batch's tests in parallel; in-level mock players are created through `com.bubbleshield.gametest.MockPlayers.createUniqueMockPlayer` (unique per-call names + automatic end-of-test PlayerList sweep on pass/fail/timeout) — never through the deprecated `GameTestHelper.makeMockServerPlayerInLevel`, whose shared "test-mock-player" name made `getPlayerByName` cross-resolve concurrent tests' mocks (the historical `whitelist_remove_revokes_uuid` flake, now fixed). New gametest classes should still declare their own `test_environment` (see `ColorGameTests.ISOLATED_ENVIRONMENT` + `data/bubbleshield/test_environment/color.json`) instead of growing the shared default batch past 50 and reshuffling it. Also beware barrier reach across the batch grid: a huge shield's `expelBlockedPlayers` sweep (e.g. `advancementMaximalist`'s diameter-200 activation, radius 100 > the ~45-block structure spacing) teleports OTHER concurrent tests' parked mock players out of their bubbles, so tests raising very large shields must run in an isolated batch and deactivate the shield before succeeding. Tests needing more room than the default 8x8x8 structure can ship an SNBT template under `data/bubbleshield/gametest/structure/*.snbt` and reference it via `@GameTest(structure = "bubbleshield:<name>")` — a minimal `{DataVersion, size, data:[one air entry], entities:[], palette:["minecraft:air"]}` works because the runner air-clears the whole `size` box before placement (see `linking_arena.snbt`, 39x8x8; keep height 8 so radius-8 shield boundary crossings stay above the barrier ceiling the framework builds at the structure top). This is the main correctness gate — use it, not just `build`. `./gradlew runServer` also boots a headless dedicated server (`run/eula.txt` is already accepted; Loom's `configureTests { eula = true }` auto-accepts the gametest EULA).
- **Shader validation (hard pre-commit gate):** the mod ships **840 generated per-effect bubble surface shaders** (`src/client/.../shaders/bubble/fx_000.fsh`–`fx_839.fsh` + the shared `surface.vsh`), **8 hand-written projector-beam shaders** (`shaders/beam/beam_storm/pulse/helix/prism/void/ember/runic/frost.fsh`, one per rendered `BeamStyle`) and **840 generated per-effect screen-fx shaders** (`src/main/.../shaders/screenfx/sfx_000.fsh`–`sfx_839.fsh`) — 1689 GLSL files total. The fx/sfx sets are emitted by deterministic byte-stable generators (all writes UTF-8 with `newline="\n"`); the workflow is: edit `tools/gen_surface_shaders.py` / `tools/gen_screen_shaders.py` / `tools/gen_post_effects.py` → regenerate (run the scripts) → `python3 tools/validate_shaders.py` → `./gradlew build` → `./gradlew runGameTest`; NEVER hand-edit generated `fx_*`/`sfx_*` files or `post_effect/effect_*.json` (regeneration overwrites them). `validate_shaders.py` is fail-CLOSED: it derives the expected file sets from `EffectRegistry.COUNT` (exactly `fx_000..fx_{COUNT-1}` + `surface.vsh`, `sfx_000..sfx_{COUNT-1}`, `post_effect/effect_00..{COUNT-1}.json` — no gaps/extras/misplaced files, recursive sweep of both shader roots) and the beam set from `BeamStyle.java`'s `RENDERED` array (cross-checked against `ShieldPipelines.java`'s `BEAM_STYLE_NAMES`), inlines the vanilla `#moj_import` includes into a per-run temp dir, runs `glslangValidator` over all 1689 files PLUS 1688 cross-stage link checks (each fx and beam shader linked against `surface.vsh`, each sfx against vanilla `core/screenquad.vsh`, with a declared-interface in/out cross-check on top since glslang's GLSL link pass misses plain varying renames) (multiprocessing; ~20 s on this VM; `--serial` fallback; the only skip is the explicit `--allow-empty` bootstrap flag — the default is fail-closed) and enforces the generated-shader invariants (byte-uniqueness, `[layer:*]` markers + `tools/surface_manifest.json` agreement, strict `FxConfig` block shape — exactly 4 `vec4` members after comment stripping — /JSON uniform-order+type+4-float-value agreement + `tools/screen_manifest.json` agreement incl. its classpath copy). Generator notes: `--only` partial runs always rewrite the FULL manifest (only file writes are restricted) and reject reversed/empty ranges; `gen_post_effects.assert_registry_in_sync` cross-checks COUNT, PARAM_CYCLE and every palette pair against `EffectRegistry.java`; the surface generator bakes accentPalette from the registry's authored primary color for every id. It must exit 0 after ANY shader change — this is a hard gate, not a nicety: every per-effect surface pipeline is registered up front and precompiled on EVERY client resource load (COUNT=840 pipelines; estimated a few seconds total, unverifiable headless), and any single invalid surface shader fails the whole client resource load with `RuntimeException("Failed to load required shader programs")`. Screen shaders compile lazily (first bubble entry) and merely log+drop on failure, but gate them identically. Gametests cannot see the client-only bubble shaders.
- **No GPU on this VM — but there IS a software-rendered screenshot harness.** `./gradlew runClient` needs real OpenGL and will not render interactively here. For actual pixels, use the client gametest harness `./gradlew runClientGameTest` (Loom's `enableClientGameTests`; `com.bubbleshield.clienttest.ShaderScreenshotTest`) under llvmpipe software GL: `DISPLAY=:1 LIBGL_ALWAYS_SOFTWARE=1 MESA_GL_VERSION_OVERRIDE=4.5` — it creates a flat world, raises a shield and writes framed bubble/beam/inside-post-fx screenshots to `/tmp/bubble_captures` (override via `BUBBLESHIELD_CAPTURE_DIR`/`_IDS`/`_BEAM`; see the class javadoc). It is slow (minutes) and llvmpipe is not a real driver, so ALSO verify shader/rendering changes by: (1) `glslangValidator` on the GLSL (installed; `tools/validate_shaders.py` inlines the `#moj_import` includes and does all of this), (2) structural diffing against vanilla 26.2 shaders, and (3) build + `runGameTest` (asset-existence + JSON-parse tests). Visual quality claims ("stunning"/"perfect"/"8K") are NOT certifiable here — llvmpipe screenshots prove geometry/compile-validity, not driver-accurate beauty. State this limitation honestly rather than overclaiming visual verification.
- **26.2 is UNOBFUSCATED / Mojang official mappings (NOT Yarn).** Old Yarn-name tutorials are wrong here (e.g. it's `Creeper`/`CreativeModeTabEvents`, not `CreeperEntity`/`ItemGroupEvents`; `AbstractArrow` is in `...projectile.arrow`). The build uses the `net.fabricmc.fabric-loom` plugin with plain `implementation` deps (no `modImplementation`/`remapJar`). When unsure of an API, decompile with `./gradlew genSources` and read the sources jars under `.gradle/loom-cache/minecraftMaven/**/*-sources.jar` — always confirm signatures there before using them.
- **26.2 API shape gotchas** (verified in this codebase): block-entity NBT uses `ValueInput`/`ValueOutput` (not `CompoundTag` params); GUIs are extractor-based (`extractBackground`/`extractLabels`, not `renderBg`); world rendering is submit-based (`LevelRenderEvents.COLLECT_SUBMITS` + `submitCustomGeometry`; `WorldRenderEvents` does not exist); datapack folders are singular (`loot_table`, `recipe`, `tags/block`).
- **Gametests live in the main source set** (`com.bubbleshield.gametest.ShieldGameTests`, registered via the `fabric-gametest` entrypoint) with `fabricApi { configureTests { createSourceSet = false; eula = true } }` in `build.gradle`. Post-effect/screenfx assets are intentionally under `src/main/resources` (not `src/client/resources`) so the headless gametest server classpath can see them for the asset-existence test.
