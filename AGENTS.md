# Bubble Shield — Agent Notes

A Fabric mod for **Minecraft 26.2** that adds a furnace-like "Bubble Shield Projector" block: insert fuel (and optional upgrade core / flux capacitor / augment items), open its GUI to whitelist players, pick 1 of **840 effects** (ids 0..839; 120 inside behaviors x 7 variants; 60 surface + 28 screen technique families), and set the bubble diameter (8–200), shape (10 shapes: sphere/dome/cylinder/cube/diamond/ring/pyramid/lens/hourglass/star), beam style (8 rendered styles — storm/pulse/helix/prism/void/ember/runic/frost — plus None/Auto) and mode (defense/pulse/eco, plus an optional periodic effect-cycle toggle). The bubble is a server-authoritative shield with per-effect surface shaders, inside behaviors, guard styles, context reactivity, ambient sounds and per-effect in-bubble screen post-effects + HUD; projectiles damage it (shrinking below 60% health) until it breaks (tiered 15/10/6/3-min cooldown). Strength mechanics: 4 core tiers (0..3) with size/tier/gamerule-scaled max HP (50..8000, `bubbleshield:strength` gamerule), tier + plating damage resistance, out-of-combat regen, augments (reinforced plating / blast ward), patch-kit repairs + emergency revive, and combat behaviors (hostile-mob barrier, arrow riposte, last stand, break nova, siege alarm + threat log). Comparator/redstone integration and advancements included. The membrane is a living system: server-broadcast visual events (`ImpactBatchS2C`) drive CPU impact waves/crest glow, last-stand tremble, whitelisted-player apertures with passage ripples, a blocked-player contact-flash HUD + layered hit-point audio, volumetric shell thickness with per-family inner materials, per-effect themed floating interiors (incl. 10 novelty rethemes) and a `bubbleshield-client.json` client config (interiorDensity / volumetricMode / flashIntensity). See `README.md` for the feature/user-facing overview and the standard `./gradlew` commands.

## Cursor Cloud specific instructions

- **Toolchain:** Minecraft 26.2 requires **JDK 25** (Temurin 25 is installed at `/usr/lib/jvm/jdk-25.0.3+9`). The startup update script installs it and persists `JAVA_HOME` in `~/.bashrc`. If a non-login shell lacks it, run `export JAVA_HOME=/usr/lib/jvm/jdk-25.0.3+9 && export PATH=$JAVA_HOME/bin:$PATH` before any `./gradlew` command. The build targets/`release`s Java 25, so an older JDK will fail.
- **First build is slow:** `./gradlew build` downloads Gradle 9.5.1 and Minecraft 26.2 on first run (several minutes); later builds are ~15–30s. There is no system-wide `gradle` — always use the wrapper `./gradlew`.

plating/blast-ward augments + quickMove (`AugmentGameTests`, env `augment`), mob barrier/riposte/last stand/nova/siege alarm + alarm rearm rate limit + threat-census identity matrix (`CombatGameTests`, env `combat`), GUI readout slots/ready ping/beacon transition cues/threat log + `/bubbleshield log`+`status` exact-output checks (`UxGameTests`, env `ux`), boss bar membership/progress/overlay round trip + shield naming (`BossBarNameGameTests`, env `bossbar`), visual-event wire format/emission sites/coalescing + hit-point sound trio/antipode tail/contact rate The vanilla runner batches 50 tests per environment and ticks a batch's tests in parallel; in-level mock players are created through `com.bubbleshield.gametest.MockPlayers.createUniqueMockPlayer` (unique per-call names + automatic end-of-test PlayerList sweep on pass/fail/timeout) — never through the deprecated `GameTestHelper.makeMockServerPlayerInLevel`, whose shared "test-mock-player" name made `getPlayerByName` cross-resolve concurrent tests' mocks (the historical `whitelist_remove_revokes_uuid` flake, now fixed). New gametest classes should still declare their own `test_environment` (see `ColorGameTests.ISOLATED_ENVIRONMENT` + `data/bubbleshield/test_environment/color.json`) instead of growing the shared default batch past 50 and reshuffling it — the default batch currently holds **47** tests (`ShieldGameTests`, `TierGameTests`, `ModeGameTests`, `GuardContextGameTests`, `ControlShapeGameTests`, plus the non-isolated halves of `AdvancementGameTests`/`EffectCatalogGameTests`); the other 21 environments under `data/bubbleshield/test_environment/` each stay well under the 50-per-batch cap. Also beware barrier reach across the batch grid: a huge shield's `expelBlockedPlayers` sweep (e.g. `advancementMaximalist`'s diameter-200 activation, radius 100 > the ~45-block structure spacing) teleports OTHER concurrent tests' parked mock players out of their bubbles, so tests raising very large shields must run in a SINGLETON batch (`advancementMaximalist` has its own `bubbleshield:advancement_maximal` environment holding only that test) and deactivate the shield before succeeding. Tests needing more room than the default 8x8x8 structure can ship an SNBT template under `data/bubbleshield/gametest/structure/*.snbt` and reference it via `@GameTest(structure = "bubbleshield:<name>")` — a minimal `{DataVersion, size, data:[one air entry], entities:[], palette:["minecraft:air"]}` works because the runner air-clears the whole `size` box before placement (see `linking_arena.snbt`, 39x8x8; keep height 8 so radius-8 shield boundary crossings stay above the barrier ceiling the framework builds at the structure top). This is the main correctness gate — use it, not just `build`. `./gradlew runServer` also boots a headless dedicated server (`run/eula.txt` is already accepted; Loom's `configureTests { eula = true }` auto-accepts the gametest EULA).
- **Shader validation (hard pre-commit gate):** the mod ships **840 generated per-effect bubble surface shaders** (`src/client/.../shaders/bubble/fx_000.fsh`–`fx_839.fsh` + the shared `surface.vsh`), **8 hand-written projector-beam shaders** (`shaders/beam/beam_storm/pulse/helix/prism/void/ember/runic/frost.fsh`, one per rendered `BeamStyle`) and **840 generated per-effect screen-fx shaders** (`src/main/.../shaders/screenfx/sfx_000.fsh`–`sfx_839.fsh`) — 1689 GLSL files total. The fx/sfx sets are emitted by deterministic byte-stable generators (all writes UTF-8 with `newline="\n"`); the workflow is: edit `tools/gen_surface_shaders.py` / `tools/gen_screen_shaders.py` / `tools/gen_post_effects.py` → regenerate (run the scripts) → `python3 tools/validate_shaders.py` → `./gradlew build` → `./gradlew runGameTest`; NEVER hand-edit generated `fx_*`/`sfx_*` files or `post_effect/effect_*.json` (regeneration overwrites them). `validate_shaders.py` is fail-CLOSED: it derives the expected file sets from `EffectRegistry.COUNT` (exactly `fx_000..fx_{COUNT-1}` + `surface.vsh`, `sfx_000..sfx_{COUNT-1}`, `post_effect/effect_00..{COUNT-1}.json` — no gaps/extras/misplaced files, recursive sweep of both shader roots) and the beam set from `BeamStyle.java`'s `RENDERED` array (cross-checked against `ShieldPipelines.java`'s `BEAM_STYLE_NAMES`), inlines the vanilla `#moj_import` includes into a per-run temp dir, runs `glslangValidator` over all 1689 files PLUS 1688 cross-stage link checks (each fx and beam shader linked against `surface.vsh`, each sfx against vanilla `core/screenquad.vsh`, with a declared-interface in/out cross-check on top since glslang's GLSL link pass misses plain varying renames) (multiprocessing; ~20 s on this VM; `--serial` fallback; the only skip is the explicit `--allow-empty` bootstrap flag — the default is fail-closed) and enforces the generated-shader invariants (code-uniqueness on the COMMENT-STRIPPED GLSL — two shaders whose executable code is identical fail even if unique-id comments differ, `[layer:*]` markers + `tools/surface_manifest.json` agreement incl. the in-file `[layer:motif:<class>:<env>]` marker matching the manifest's per-id motif fingerprint, the v11 cost lint (`check_cost_budget`: exactly **twelve** texture taps per fx — 7 `atlasTile` + 3 `Sampler1` + 2 `Sampler2` — field-call/line ceilings, and the `[layer:thick:*]`/`[layer:inner:*]` volumetric-thickness markers matching the manifest's `thick`/`costTaps`/`costField`), strict `FxConfig` block shape — exactly 4 `vec4` members after comment stripping — /JSON uniform-order+type+4-float-value agreement + `tools/screen_manifest.json` agreement incl. its classpath copy). Generator notes: `--only` partial runs always rewrite the FULL manifest (only file writes are restricted) and reject reversed/empty ranges; `gen_post_effects.assert_registry_in_sync` cross-checks COUNT, PARAM_CYCLE and every palette pair against `EffectRegistry.java`; the surface generator bakes accentPalette from the registry's authored primary color for every id. It must exit 0 after ANY shader change — this is a hard gate, not a nicety: every per-effect surface pipeline is registered up front and precompiled on EVERY client resource load (COUNT=840 pipelines; estimated a few seconds total, unverifiable headless), and any single invalid surface shader fails the whole client resource load with `RuntimeException("Failed to load required shader programs")`. Screen shaders compile lazily (first bubble entry) and merely log+drop on failure, but gate them identically. Gametests cannot see the client-only bubble shaders.
- **Surface shaders sample a real TEXTURE ATLAS + the SCENE COPY (not purely procedural).** Every bubble `fx_*.fsh` samples `uniform sampler2D Sampler0` = one shared 4096x2048 32-tile (8x4 grid of 512px tiles) atlas at `src/main/resources/assets/bubbleshield/textures/effect/surface_atlas.png` (+ `.png.mcmeta`, LINEAR/REPEAT), generated deterministically by `tools/gen_textures.py` (RGB = coarse/mid/fine structural detail, A = emission/glow mask; recolor-neutral grayscale so the per-effect palette still tints everything), PLUS `Sampler1`/`Sampler2` = the `SceneCopy` scene color/depth copy (blitted after the solid pass, only when a shield's AABB intersects the camera frustum): the surfaces now REFRACT the world behind the membrane (chromatic three-tap offset with a depth guard), carry fake key LIGHTING from the atlas height gradient, and FLOW (two-phase curl-advected atlas crossfade) — they are refractive energy glass, not flat decals. Binding: `ShieldPipelines.BUBBLE_SNIPPET` adds `BindGroupLayouts.SAMPLER0_SAMPLER1_SAMPLER2` and each render type does `RenderSetup.builder(pipeline).withTexture("Sampler0", id("textures/effect/surface_atlas.png"))` + `.withTexture("Sampler1"/"Sampler2", SceneCopy.SCENE_COLOR_ID/SCENE_DEPTH_ID, ...)` — the sampler NAMEs must match in all three places (bind group / GLSL uniform / `withTexture`) or all 840 pipelines fail at client resource load. `validate_shaders.py` hard-gates this contract (`check_texture_binding`): every fx must declare AND use `Sampler0`, the `Sampler1`/`Sampler2` pair is both-or-neither per file (declaring/sampling one without the other is a hard failure — SceneCopy arms color+depth together), no `layout(binding=...)` (illegal at #version 330), beams stay SamplerN-free, and the atlas PNG must exist + be 4096x2048 RGBA and fully decode (chunk CRCs, IEND, IDAT inflates to the exact raster size). The surface shaders read near-opaque where refraction is active (what shows through must be the refracted copy), but the whitelisted-player dissolve still wins because the FINAL alpha is `vertexColor.a * clamp(bodyAlpha, 0, aMax)` with the `vertexColor.a` multiply as the last op — keep it that way. If you edit `tools/gen_textures.py`, regenerate the atlas AND re-run `validate_shaders.py`.
- **No GPU on this VM — but there IS a software-rendered screenshot harness.** `./gradlew runClient` needs real OpenGL and will not render interactively here. For actual pixels, use the client gametest harness `./gradlew runClientGameTest` (Loom's `enableClientGameTests`; `com.bubbleshield.clienttest.ShaderScreenshotTest`) under llvmpipe software GL: `DISPLAY=:1 LIBGL_ALWAYS_SOFTWARE=1 MESA_GL_VERSION_OVERRIDE=4.5` — it creates a flat world, raises a shield and writes framed bubble/beam/inside-post-fx screenshots to `/tmp/bubble_captures` (override via `BUBBLESHIELD_CAPTURE_DIR`/`BUBBLESHIELD_CAPTURE_IDS`/`BUBBLESHIELD_CAPTURE_BEAM`; see the class javadoc). Two sibling harnesses in the same entrypoint no-op unless enabled by env var: `ScenarioCaptureTest` (`BUBBLESHIELD_SCENARIOS=dynamics,interiors,volumetric,contact`, any subset — staged shots of the dynamic systems: `impact_wave_t0/t1/t2` via a real `queueImpact`+`applyShieldDamage`, `aperture_open`/`passage_close` hysteresis pair, novelty interiors 633/839/809 inside + through-membrane, `volumetric_grazing` limb view, and `contact_flash` — that one re-owns the shield to a foreign UUID + clears BOTH whitelist sets + `markUpdated()`, teleports to the wall and HOLDS the walk-backward key through the capture so the predicted-press trigger fires (the &gt;2-block "snap" trigger requires the PRE-snap position within 1.5 blocks of the wall, so ordinary staging teleports no longer count — only the barrier's own expulsion teleports do), gated on the live `ContactFlash.alpha()`; the flash envelopes are wall-clock millis while gametest client ticks run much faster than 50 ms, so rate-limit clearing waits must be wall-clock, not tick-counted) and `VideoCaptureTest` (`BUBBLESHIELD_VIDEO=1` — per-tick frame clips incl. `aperture_walkthrough` and the 3-hit `impact_volley`; `BUBBLESHIELD_VIDEO_CLIPS=<name substrings>` restricts the clip set; encode with `python3 tools/encode_captures.py`). It is slow (minutes) and llvmpipe is not a real driver, so ALSO verify shader/rendering changes by: (1) `glslangValidator` on the GLSL (installed: 15.4.0 at `/usr/local/bin/glslangValidator`, built from the KhronosGroup source tarball because apt and the GitHub-release CDN are egress-blocked here; `tools/validate_shaders.py` inlines the `#moj_import` includes and does all of this), (2) structural diffing against vanilla 26.2 shaders, and (3) build + `runGameTest` (asset-existence + JSON-parse tests). Visual quality claims ("stunning"/"perfect"/"8K") are NOT certifiable here — llvmpipe screenshots prove geometry/compile-validity, not driver-accurate beauty. State this limitation honestly rather than overclaiming visual verification.
- **Interior sprite pipeline:** the two interior sheets under `src/main/resources/assets/bubbleshield/textures/interior/` (`interior_pixel.png` 8x8x64px NEAREST, `interior_soft.png` 4x4x128px LINEAR+clamp) are written by the DETERMINISTIC assembler `python3 tools/gen_interior_sprites.py` — byte-stable, safe to re-run. BOTH sheets carry BINARY alpha (0/255): the interior pipelines are non-blending cutout (discard only at alpha == 0), so the soft sheet's procedural falloffs are 4x4-Bayer-dithered into a screen-door dissolve (continuous alpha would render as solid squares). Its inputs under `tools/interior_sprites/raw/` are frozen one-off AI outputs that can NOT be regenerated: never delete or "improve" them, and never hand-edit the emitted sheets (rerun the assembler instead). `InteriorGameTests` pins the sheet dims + full-IDAT decodability, the soft sheet's binary alpha + blur/clamp mcmeta (and the pixel sheet's LACK of an mcmeta), and the exact 10-id novelty override set.
- **Visual-event payload contract:** `ImpactBatchS2C` carries at most 8 `ImpactEntry` rows per shield per tick (kinds IMPACT/BREAK/HEAL/CONTACT/PASSAGE_IN/PASSAGE_OUT; dir = outward unit byte-quantized). The strength byte is the pre-encoding value clamped to 25.5 then x10: **x10 of the applied damage for IMPACT, x10 of the healed HP for HEAL, the fixed sentinel 255 for BREAK (25.5 x 10 — read it back UNSIGNED via `strengthUnsigned()`, a raw byte read sign-flips it), and a fixed 2.5 -> byte 25 for CONTACT and PASSAGE_IN/OUT** (their strengths are bookkeeping, not damage scales — the client substitutes per-kind effective wave strengths). Batches are coalesced by `BubbleShieldBlockEntity.flushImpacts` which runs right AFTER `flushSync` and deliberately OUTSIDE its diff gate (pure events never change the replicated snapshot — routing them through the sync would eat them). Receivers are players within currentRadius+32 of the center. Rate limits to keep in mind: CONTACT once per player per 10 ticks (`tryContact`), one hit-point sound trio per shield per tick (`tryImpactSounds`), at most 4 queued antipode tails. `queueImpact` is public — it is the sanctioned hook for gametests and capture harnesses.
- **26.2 is UNOBFUSCATED / Mojang official mappings (NOT Yarn).** Old Yarn-name tutorials are wrong here (e.g. it's `Creeper`/`CreativeModeTabEvents`, not `CreeperEntity`/`ItemGroupEvents`; `AbstractArrow` is in `...projectile.arrow`). The build uses the `net.fabricmc.fabric-loom` plugin with plain `implementation` deps (no `modImplementation`/`remapJar`). When unsure of an API, decompile with `./gradlew genSources` and read the sources jars under `.gradle/loom-cache/minecraftMaven/**/*-sources.jar` — always confirm signatures there before using them.
- **26.2 API shape gotchas** (verified in this codebase): block-entity NBT uses `ValueInput`/`ValueOutput` (not `CompoundTag` params); GUIs are extractor-based (`extractBackground`/`extractLabels`, not `renderBg`); world rendering is submit-based (`LevelRenderEvents.COLLECT_SUBMITS` + `submitCustomGeometry`; `WorldRenderEvents` does not exist); datapack folders are singular (`loot_table`, `recipe`, `tags/block`).
- **Frozen menu layout + canonical damage pipeline:** `BubbleShieldMenu`'s `ContainerData` slot indices (0..19: fuel, permille health, ..., strength, threat, then the APPENDED 18 = revive-available flag and 19 = whole current HP for the GUI's exact "HP: cur/max" row — the permille slot stays for ratio/progress uses) and item-slot order (FUEL 0 / CORE 1 / CAPACITOR 2 / AUGMENT 3, player inv 4..30, hotbar 31..39) are FROZEN — data slots replicate as 16-bit signed values, so keep every encoding within [0, 32767] and only APPEND new slots, never renumber. Shield damage always flows `blastWardedDamage` (explosive x0.4) → `appliedDamage` (tier DR x plating DR multiplicative, combined cap 0.70) → last-stand x0.5 (<25% HP); resonance links split the RAW damage first, then each shield applies its own pipeline.
- **Gametests live in the main source set** (`com.bubbleshield.gametest.ShieldGameTests`, registered via the `fabric-gametest` entrypoint) with `fabricApi { configureTests { createSourceSet = false; eula = true } }` in `build.gradle`. Post-effect/screenfx assets are intentionally under `src/main/resources` (not `src/client/resources`) so the headless gametest server classpath can see them for the asset-existence test.
